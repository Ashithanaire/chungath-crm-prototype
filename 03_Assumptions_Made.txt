4. WORKFLOW & PROCESS INTERPRETATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Lead Lifecycle Interpretation:

Step 1: Lead Source Identification
├─ Assumption: Captured during lead creation
├─ Referred By field (dropdown): Walk-in | Staff | Event | Other
└─ In prototype: Dropdown in Lead Creation Form

Step 2: Lead Capture & Entry
├─ Assumption: Can happen via mobile (pending approval) OR admin panel (immediate)
├─ In prototype: Single form for both; approval workflow modeled
└─ Status behavior: Executive-created → "Pending Manager Approval" (simulated)

Step 3: Manager Approval
├─ Assumption: Modal workflow for Manager to review leads
├─ In prototype: Not explicitly shown (ready for implementation)
├─ Suggested implementation: List of "Pending Approval" leads in Manager dashboard

Step 4: Lead Assignment
├─ Assumption: Manager creates team (group of executives) → assigns to team/area
├─ Area = Pincode-based (auto-derived)
├─ In prototype: Shown as "Assigned To" field in lead detail
└─ Not fully interactive in prototype (ready for add of assignment modal)

Step 5: Lead Qualification & Follow-up
├─ Assumption: Each follow-up is a separate record (Visit to Lead)
├─ Status flow: New → Contacted → Interested / Not Interested
├─ In prototype: Follow-up Modal records status change + notes
└─ History displayed in "Follow-up History" section

Step 6: Lead Conversion
├─ Assumption: When lead commits to purchase, convert to Customer Account + Contact
├─ In prototype: Button on Lead Detail to trigger conversion (modal ready)
└─ Note: In real Salesforce, creates related Account & Contact records

Step 7: Lead Completion Request
├─ Assumption: Executive marks Converted/Lost with mandatory narration
├─ In prototype: Modal form with Status dropdown + Notes
└─ Manager then approves/rejects the completion request

Step 8: Manager Approval of Completion
├─ Assumption: If rejected, lead reverts to previous status with Manager's comments
├─ In prototype: Approval workflow modeled (ready for full implementation)
└─ UI shows: Lead status, approval status, Manager comment


Customer & Account Management Interpretation:

Creation:
├─ Assumption: Automatic on Lead Conversion
├─ Customer fields inherited from converted Lead
├─ New field: Customer Card Number (mandatory during conversion)
└─ In prototype: Customer created post-conversion (shown in Customers list)

Area Auto-Derivation:
├─ Assumption: Area auto-derived from Pincode
├─ Manager can override (in prototype, shown as editable field)
└─ Used for pincode-based area assignment (not hoarding routes)

Visit History:
├─ Assumption: Auto-populated from Field Visit records
├─ Each check-in/check-out creates a Visit record
├─ In prototype: Visit History shown in Customer Detail with GPS, photos, notes


Field Visit Workflow Interpretation:

Daily Flow:
├─ [Start Day] → Daily Log created, GPS captured
├─ [Visit 1 Check-in] → GPS captured
├─ [Visit 1 Check-out] → GPS + Photo captured
├─ [Visit 2 Check-in] → GPS captured
├─ ... (repeat for each visit)
├─ [End Day] → Daily Log completed, Google Maps API call triggered
└─ Result: Daily KM calculated & stored

Distance Matrix API Integration:
├─ Assumption: API call happens at End Day (not real-time)
├─ GPS Sequence: Start Day GPS → Visit 1 Check-in → Visit 1 Check-out → ... → End Day GPS
├─ API returns: Total road distance (KMs)
├─ Storage: Per Executive, per Day
└─ In prototype: Simulated API call; monthly KM total displayed on dashboard

Photo Restrictions:
├─ Assumption: Camera-only capture by default (mobile device camera)
├─ Gallery upload: Blocked, requires Manager request → Admin approval
├─ In prototype: Camera icon shown; gallery upload marked as approval-needed
└─ Real implementation: Client-side media picker restriction


Hoarding Management Workflow Interpretation:

Installation (Step 1-3):
├─ Marketing Executive creates Hoarding record with all master fields
├─ Size fields (Length, Width) → Auto-calculate Area (Sq.Ft)
├─ Inspection Frequency (30/60/90 days) → Auto-compute Next Visit Schedule Date
├─ Installation photo capture (camera-only, mandatory)
└─ GPS auto-captured at installation (within 50-meter accuracy)

Periodic Inspection (Step 4-7):
├─ System sends reminder on Next Visit Schedule Date
├─ Executive visits hoarding, checks in with GPS, captures photo
├─ Executive updates Board Status:
│  ├─ Good Condition
│  ├─ Need Fix (with sub-options: Vegetation, Flex Damage, Competitor Hoarding)
│  ├─ Rectified
│  └─ Others (requires mandatory note)
├─ Status note logged for audit trail (with date & executor)
└─ If visit after due date → Flagged as "Out of Due Date"

Contract Monitoring (Step 8):
├─ Assumption: System sends alert 30 days before contract expiry
├─ Alert includes: Contract end date, contract value, next steps
└─ In prototype: Alert shown in Hoarding Detail + Notifications

Lifecycle Expiry (Step 9):
├─ Assumption: After 1 year from "Active From Date", hoarding auto-marks as Inactive
├─ Removed from contractor, master, and route active lists
├─ Admin reactivates after re-installation (resets 1-year cycle)
├─ In prototype: Status shows "Inactive" after 1-year period (simulated with logic-ready code)


4. NOTIFICATIONS & ALERTS INTERPRETATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ASSUMPTION: All notifications are delivered via 3 channels:
├─ In-App Notification Bell (with count badge)
├─ Email (background, not shown in prototype)
└─ Pop-up (browser notification, ready for implementation)

Alert Types Modeled in Prototype:

Next Visit Date Reminder:
├─ Recipient: Assigned Sales Executive
├─ Trigger: On scheduled next visit date for a lead/customer
├─ Display: In-app notification, email optional
└─ In prototype: Notification bell shows count; content ready

Pending Lead Follow-up Reminder:
├─ Recipient: Sales Executive + Manager
├─ Trigger: When lead has no follow-up for configured threshold (e.g., 5 days)
├─ In prototype: Manager dashboard would show "Pending Follow-ups"

Hoarding Next Visit Alert:
├─ Recipient: Assigned Marketing Executive
├─ Trigger: On computed Next Visit Schedule Date
└─ In prototype: Hoarding Detail shows "Next Visit Due" date with visual highlight

Damaged Hoarding Follow-up:
├─ Recipient: Assigned Executive + Manager
├─ Trigger: After Board Status = "Need Fix" and not yet "Rectified"
├─ Escalation: Multiple reminders if not addressed
└─ In prototype: Hoarding status color-coded (red for damaged)

Contract Expiry Alert:
├─ Recipient: Manager + Admin
├─ Trigger: 30 days before contract end date
├─ In prototype: Shown in Hoarding Detail card highlight

Lead Completion Approval:
├─ Recipient: Manager
├─ Trigger: When Executive marks lead Converted/Lost with narration
├─ In prototype: Modal form to submit completion; Manager approval pending

Lead Creation Approval:
├─ Recipient: Manager
├─ Trigger: When Sales Executive creates lead via mobile (pending approval)
├─ Status: Lead enters "Pending Manager Approval" state
└─ In prototype: Status shown in Lead List

Ceremony Date Approaching:
├─ Recipient: Assigned Sales Executive
├─ Trigger: Configurable days before ceremony date (e.g., 7 days)
├─ Purpose: Reminder to follow up before ceremony
└─ In prototype: Ready for implementation in Lead Detail

Upcoming Event Reminder:
├─ Recipient: Manager + Admin
├─ Trigger: Configurable days before event/campaign start (default: 7 days)
├─ In prototype: Event Detail would show reminder configuration


5. DASHBOARD & REPORTS INTERPRETATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Dashboard Metrics Interpretation:

Admin Dashboard:
├─ Total Leads: Count of all leads (all branches, all statuses)
├─ Leads Converted: Count of leads with status = "Converted"
├─ Active Hoardings: Count of hoardings with Contract Status = "Active"
└─ Avg KMs/Day: Sum of all executive KMs / number of field days

Manager Dashboard:
├─ Branch-level metrics (same as Admin but filtered by manager's branch)
├─ Team Performance: Leads per executive, conversion rate, KMs
└─ Lead Approvals Pending: Count of leads awaiting manager approval

Executive Dashboard:
├─ Personal metrics only
├─ Today's Visits: Count of visits scheduled for today
├─ KMs Today: Running total (updated at End Day)
└─ Conversion Rate: Personal conversion rate for this month

Reports Interpretation:

Sales Productivity Reports:
├─ Total Visits / Completed Visits: Daily and Monthly aggregations
├─ Status-wise count: Converted vs Lost (success metrics)
├─ No Visit Days: Days when executive didn't check in
├─ Start Day vs End Day: Consistency tracking (should be 1:1)

Lead & Customer Reports:
├─ Leads Collected: Count by day/month
├─ Lead Conversion Rate: (Converted) / (Total) * 100
├─ New Customer Onboarding: Count of newly converted customers
├─ Leads by Source: Breakdown by Walk-in, Staff, Event, Other

Hoarding Monitoring Reports:
├─ Number of Visits: Count of inspection visits per month
├─ Missed Visits: Inspections that were "Out of Due Date"
├─ Board Status Summary: Count by status (Good, Damaged, Rectified, etc.)
└─ Visit History with Photos: Excel export with GPS-tagged photos & notes

Contractor Reports:
├─ Annual Contract Value: Sum of (Annual Rate) per contractor, per route
├─ Active Hoardings: Count of active hoardings per contractor
├─ Contract Expiry Soon: Contracts ending in 30/60/90 days
└─ Sq.Ft Summary: Total area of active hoardings

8. FIELD VISIT TRACKING ASSUMPTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GPS Accuracy:
├─ Installation: Within 50 meters of actual hoarding location
├─ Check-in/Check-out: Standard mobile GPS accuracy
└─ Requirement: Sufficient for distance calculation & mapping

Google Maps Distance Matrix API:
├─ Called once per day (at End Day)
├─ Input: GPS coordinates sequence (Start → Visits → End)
├─ Output: Road distance (KMs) between consecutive points
├─ Cost: Within free tier (70,000 requests/month) at 4 executives
└─ Billed to: Chungath Jewellers (on their Google Cloud account)

Photo Capture:
├─ Camera-only by default (mobile device camera)
├─ Gallery upload: Requires Manager request → Admin approval
├─ Photo metadata: GPS-tagged with timestamp, executor name
├─ Storage: Cloud-based (Salesforce Files or external cloud)
└─ In prototype: Photo capture action shown; real storage ready for Salesforce Files API

Accuracy Requirements:
├─ Visit time: ± 1 minute (device time)
├─ GPS coordinates: Decimal format (9.9312°N, 76.2673°E)
└─ Distance: Calculated to nearest 0.1 KM

12. DESIGN DECISIONS EXPLAINED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Modal vs Full-Screen:
├─ Decision: Forms (Create, Edit, Record) use Modal overlays
├─ Reason: Non-disruptive, maintains context of parent page
├─ Alternative: Full-screen form pages (ready for implementation)
└─ SLDS Pattern: Consistent with Salesforce modal dialogs

Two-Column Detail Layout:
├─ Decision: Main content (left 2fr) + Sidebar (right 1fr)
├─ Reason: Action buttons always visible while scrolling content
├─ Alternative: Single column (mobile) (responsive code ready)
└─ SLDS Pattern: Record pages with sidebar actions

Alternating Row Colors:
├─ Decision: Tables use alternating background colors (ALT1/ALT2)
├─ Reason: Improves readability in long tables
├─ Implementation: Conditional background based on row index
└─ SLDS Pattern: Standard data table styling

Collapsible Sidebar:
├─ Decision: Sidebar can collapse to icons
├─ Reason: Maximize content area on screens with many columns
├─ Implementation: Width transition with conditional label display
└─ SLDS Pattern: Common navigation pattern

Status Badges with Color-Coding:
├─ Decision: Status displayed as colored pill badges
├─ Colors: Green (success), Yellow (warning), Red (error), Gray (neutral)
├─ Reason: Quick visual recognition of status at a glance
└─ SLDS Pattern: Status badge component

KPI Cards:
├─ Decision: Dashboard uses card grid for metrics
├─ Information: Metric name, value, trend percentage
├─ Color: Metric-specific (success green, primary blue, etc.)
└─ SLDS Pattern: Card grid layout

History as Chronological Items:
├─ Decision: Follow-up, inspection, visit history shown as expandable items
├─ Content: Date, action, notes, related data, status
├─ Styling: Light background, subtle styling (de-emphasized)
└─ SLDS Pattern: Activity timeline/feed
